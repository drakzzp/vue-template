Next.js 高频面试 10 题（这家短剧公司必考，精简背诵版）
1. Next 和原生 React 区别？
原生 React 客户端渲染，返回空 HTML，首屏慢、爬虫抓取不到内容；Next 内置 SSG/SSR/ISR 多种渲染方案，服务端生成完整 HTML，首屏快、SEO 友好，自带文件路由、图片优化、API 路由，适配海外短剧站点谷歌收录需求。
2. SSG、SSR、ISR 区别，项目怎么选型？
SSG：构建时一次性生成静态 HTML，放 CDN，速度最快，用于首页、固定分类页；
SSR：每次访问服务端实时请求数据渲染，数据实时，用于个人中心；
ISR：预生成页面，配置定时缓存失效，过期重新生成，短剧列表、剧集详情首选，兼顾速度与内容更新。
3. App Router 与 Pages Router 区别？
Pages：pages 目录，getStaticProps 等方法拿数据，组件默认客户端；
App：app 目录，默认 RSC 服务端组件，async 直接 await 请求数据，原生嵌套路由、loading/error 边界、metadata 自动配置 SEO，新项目主流。
4. RSC 服务端组件和 'use client' 区别？
RSC：运行在服务端，不能用 hooks、浏览器 API，不打包 JS 到前端，页面展示、列表全用它，减小首屏体积；
'use client'：转为客户端组件，可写交互逻辑、事件，仅收藏、弹窗等交互模块使用。
5. next/image 作用？
自动图片压缩、格式优化、懒加载、自适应尺寸，优化 LCP，短剧页面大图多，大幅提升页面性能与谷歌评分。
6. Next Middleware 用处？
运行在请求阶段，可做多语言路由跳转、域名重定向、访问鉴权，适配海外多地区、多语种业务。
7. Next 怎么做 SEO 优化？
合理选用 ISR/SSG 保障爬虫拿到完整 HTML；
metadata 配置标题、描述；
生成 sitemap.xml；
next/image 优化图片指标；
Middleware 处理 301 重定向。
8. Next API Route 是什么？
在 app/api 下创建接口，Node 环境运行，可充当后端接口，前端直接调用，项目简易接口无需单独搭后端。
9. 部署 Next 项目有哪些方式？
Vercel 一键部署、服务器 Docker 部署、Nginx 反向代理部署。
10. 首屏性能优化做过哪些？
合理拆分 RSC 与客户端组件、ISR 缓存、图片优化、静态资源 CDN、路由按需加载、控制客户端打包体积。

Next四层缓存架构图
一、四层缓存（从快→慢，面试必说）
1. Request Memoization（请求记忆，单次渲染周期）
作用：同一次页面渲染，多个组件同 URL 只发 1 次请求，自动去重
生命周期：仅当前请求内生效，跨用户、跨请求不共享，默认自动开启无需配置。
例：layout 和 page 同时请求首页 banner，框架合并成一次接口调用。
2. Data Cache（服务端持久数据缓存，核心）
存储 fetch/db 结果，跨用户、跨请求复用，持久化缓存，由cache/revalidate/tags控制规则。
3 种 fetch 配置（项目分场景使用）：
tsx
//① SSG永久缓存（默认force-cache）：资讯、帮助页、固定导航
fetch(url, { cache: "force-cache" }) 

//② ISR定时增量缓存：商品、文章列表，60s过期，过期后访问先返回旧数据、后台异步更新
fetch(url, { next: { revalidate: 60 } }) 

//③ 禁用缓存SSR：个人中心、购物车、实时价格，每次请求实时查源
fetch(url, { cache: "no-store" })
补充：数据库查询缓存用 unstable_cache（ORM/Prisma 无 fetch 场景）
tsx
import { unstable_cache } from "next/cache"
const getList = unstable_cache(async()=>db.query(),["goods"],{revalidate:300})
3. Full Route Cache（整页 HTML/RSC 载荷缓存）
缓存整页面渲染产物（HTML+RSC 数据），SSG/ISR 页面构建 / 首次访问后静态固化，CDN 可缓存整页，极致优化 SEO 与首屏 TTFB。
SSG：build 打包生成静态 html，永久缓存
ISR：revalidate 到期后后台重新生成新页面缓存
4. Router Cache（客户端浏览器缓存）
前端 SPA 路由跳转缓存页面数据，前进 / 后退不发请求，默认缓存 30s，可通过revalidatePath清空路由缓存。
二、两种数据预存方案（预取优化首屏）
1. 服务端预取（RSC 天然预存，最优 SEO）
并行预请求：提前在外层定义请求 Promise，组件内 await，多个接口并行发起、缩短等待
tsx
//页面顶层提前发起请求，并行预取
const listP = fetch("/api/list",{next:{revalidate:30}})
const bannerP = fetch("/api/banner",{next:{revalidate:60}})
export default async function Page(){
  const [list,banner]=await Promise.all([listP,bannerP])
}
generateStaticParams 构建预生成动态路由：商品详情 / 文章详情打包时预渲染批量页面，上线直接走静态缓存。
tsx
//打包预生成100篇文章静态页面
export async function generateStaticParams(){
  const arr=await fetch("/api/posts").then(r=>r.json())
  return arr.map(item=>({slug:item.slug}))
}
Suspense 流式预渲染：loading.tsx+Suspense 分片渲染，优先返回骨架，剩余内容后台逐步预取推送，优化 LCP。
2. 客户端预存（use client 组件，交互数据）
路由预加载：prefetch={true}，Link 组件悬浮提前预拉取下一页数据
tsx
<Link href="/detail" prefetch>商品详情</Link>
SWR/ReactQuery 内存缓存：客户端接口数据本地缓存，短时间重复点击不重复请求；配合 localStorage 持久化用户筛选条件。
三、手动缓存失效（事件驱动，业务数据更新立刻清缓存，面试高频）
1. 按路径失效 revalidatePath（单页面 / 整目录刷新）
tsx
'use server'
async function updateGoods(){
  await db.update()
  revalidatePath("/goods") //刷新商品列表全页缓存
  revalidatePath("/goods/[id]") //刷新详情动态路由
}
2. 标签批量失效 revalidateTag（推荐，统一管理一类数据）
tsx
//存数据时打tag
fetch("/goods",{next:{tags:["goods"]}})
//修改后一键清空所有带goods标签的缓存
revalidateTag("goods")
适用：改分类后，所有关联商品页面一次性刷新缓存，不用逐个路径配置。
四、项目落地分层策略（面试项目描述原话）
静态页面（帮助、公司介绍）：force-cache SSG，打包预生成，永久缓存，CDN 加速，SEO 最优
内容半更新（商品、资讯）：revalidate ISR，定时过期 + 修改后手动 revalidateTag，兼顾速度与新鲜度
个性化实时页（个人中心、订单）：no-store SSR，每次实时拉取用户数据，禁用缓存
弹窗、筛选等客户端组件：SWR 本地缓存 + Link 预取，提升操作流畅度
五、精简口述版（面试直接说 1 分钟）
我们项目基于 Next14 App Router 四层缓存体系，数据分三级缓存配置：固定静态内容用 force-cache 全静态 SSG，商品资讯用 revalidate 做 ISR 增量缓存，用户个性化页面 no-store 走实时 SSR；服务端通过提前 Promise 并行预取 + generateStaticParams 打包预生成动态页面优化首屏；数据在 Server Action 更新后用 tag 批量失效缓存；客户端靠 Link 预加载 + SWR 做前端数据预存，兼顾 SEO、首屏性能和数据实时性。

极简大白话：水合 (Hydrate)
SSR/ISR 流程两步：1. 服务端出完整 HTML 给浏览器 → 2. 前端 JS 运行、绑定事件 = 水合
1. 过程
浏览器收到后端返回现成 HTML，页面立刻能看见内容（按钮、文字全都出来），但是按钮点不动、没有交互
等客户端 JS 下载完毕执行，React 把 DOM 和组件实例绑定、挂载点击 / 输入事件，页面从 “纯静态看的页面” 变成可交互页面，这个绑定动作就叫水合
2.Next 两个关键点
Pages 全页面需要整页水合：整页 JS 下载完才批量水合，容易出现白屏能看见、点击没反应（水合断层）
App Router (RSC) 碎片化水合：只有use client的组件才下发 JS、局部水合；页面主体 RSC 无客户端 JS，不用水合，首屏更快
3. 面试一句话
水合就是浏览器拿到 SSR 的静态 HTML 后，客户端 JS 挂载 DOM 事件、激活页面交互的过程。

水合最容易出的 4 个问题（面试必问、背这 4 条就够）
我用最简单、最真实、面试官最爱听的话讲，不绕弯、不装专业。
1. 水合不匹配（最常见！Hydration Mismatch）
就是：
服务端渲染的 HTML 和客户端渲染的不一样 → 直接报错！
常见原因：
页面里写了 Date.now() / Math.random()
用了 window / document（服务端没有）
前端判断设备、屏幕宽度，服务端不知道
组件条件渲染前后不一样
一句话背：
服务端出的 HTML 和前端水合时结构不一致，导致水合失败报错。
2. 水合卡顿、页面点不动
SSR 页面：
先看到内容（能看）
但要等 JS 下载 + 水合完才能点击
水合慢 → 按钮点不动、输入框输不了
尤其 Pages Router 整页水合最明显。
一句话背：
水合期间 JS 阻塞，页面可看不可点，体验差。
3. 首屏 JS 太大，水合慢
Pages Router 所有组件都是客户端组件
→ JS 体积大
→ 下载慢
→ 水合慢
→ LCP 变差、SEO 受影响
一句话背：
大量客户端 JS 导致水合延迟，性能指标下降。
4. useEffect 导致内容闪烁
水合完成后 → useEffect 执行 → 数据更新 → 页面重新渲染
→ 内容闪一下
用户体验很差。
一句话背：
水合完成后触发重渲染，导致页面闪烁。

useEffect 闪烁：水合完重渲染导致页面跳动。

Server Component 是在服务端运行的，用户请求过来后，服务端直接渲染组件生成 HTML 返回到浏览器，浏览器直接展示内容，不发送任何 JS，不需要水合。
而 Client Component 会把 JS 发送到浏览器，需要下载、解析、执行、水合之后才能交互。
最大区别就是：RSC 只在服务端渲染，无 JS 无交互；Client Component 在浏览器渲染，有 JS 有水合，支持交互。

四、RSC 和 Client Component 最核心区别（面试必背）
1. 运行地方不一样
RSC：服务器 (Node) 运行
Client：浏览器运行
2. 会不会发 JS 到浏览器
RSC：完全不发 JS
Client：要发 JS，要下载
3. 能不能取数据
RSC：直接 async/await、fetch、查数据库
Client：不能直接查库，只能发接口
4. 能不能用交互 API
RSC：不能用 useState、useEffect、onClick
Client：可以用所有交互
5. 需不需要水合
RSC：不需要水合
Client：必须水合才能点击
6. 性能
RSC：JS 体积几乎为 0 → 超快
Client：JS 大 → 慢、水合久


结论：Client 复用 RSC 缓存的 4 种方案，完全依托 Next 四层缓存架构，每一种方案对应一层缓存
四层缓存清单（从上到下：客户端→服务端）：
Router Cache（浏览器内存，客户端层）：缓存 RSC 渲染后的 Payload
Full Route Cache（服务端磁盘）：缓存整页 HTML+RSC Payload
Data Cache（服务端磁盘，跨用户持久）：缓存fetch/db原始数据（RSC 核心缓存）
Request Memoization（单次请求内存，React.cache）：同一次页面渲染内去重请求


四层缓存金字塔
一、方案 1：RSC 查数据→Props 传给 Client → 复用【Request Memoization + Data Cache】
tsx
// RSC预取，自动进入Data Cache+单次Request缓存
const list = await getList()
<Client list={list}/>
Request Memoization：当前页面渲染，RSC 多处调用同一个查询只查 1 次（同请求内存缓存）；
Data Cache：查询结果落地服务端全局缓存，下次任何人进页面不用重复查库；
Client 直接拿序列化后数据，完全不走浏览器 fetch，天然复用两层服务端缓存。
对应：复用 3+4 层缓存
二、方案 2：unstable_cache + Client use () → 精准复用【Data Cache（第 3 层）】
tsx
// lib统一封装，存入全局Data Cache
export const getGoods = unstable_cache(fn,['key'],{revalidate:xxx})
// Client use(getGoods(id))
unstable_cache：绑定 Data Cache（跨请求持久缓存），RSC 提前调用写入缓存；
Client 里use(异步函数)不会走浏览器原生 fetch，请求中转 Next 服务端，优先命中已存在的 Data Cache；
原生fetch('/第三方接口')是浏览器直连，绕开 Next 缓存层，无法命中 Data Cache（这就是普通 client fetch 拿不到 RSC 缓存的本质）。
对应：复用第 3 层 Data Cache
三、方案 3：React.cache + Context 传 Promise → 复用【Request Memoization（第 4 层）】
tsx
// React.cache = Request Memoization，仅限单次请求
export const getUser = cache(async()=>db.query())
// RSC拿到Promise传给Context，Client use(promise)
React.cache就是Request Memoization 实现：** 仅限同一次页面刷新（同一个服务端请求生命周期）** 缓存，页面刷新即失效，不落地 Data Cache；
RSC 生成 Promise，透传到 Client 组件，use()直接读取已经执行完的缓存结果，同页面所有 Client 组件共享一次数据库请求。
对应：复用第 4 层单次请求内存缓存
四、方案 4：Client 请求本项目 /api 路由 → 复用【Data Cache + Full Route Cache】
tsx
// api/route.ts内部RSC风格fetch，自动走Data Cache
export async function GET(){
  const data = await fetch(url,{next:{tags:['goods']}})
}
// Client fetch('/api/goods')
/api/*运行在 Next 服务端，内部 fetch 受 Next 缓存管控，和页面 RSC 共用同一个全局 Data Cache；
静态接口结果还会存入 Full Route Cache（整接口响应缓存）；
Client 请求同域名 API = 经过 Next 缓存层，复用 RSC 提前缓存的数据。
对应：复用第 3+2 层缓存
五、补充：Router Cache（第 1 层客户端缓存）影响
Router Cache 只缓存页面最终 RSC Payload（渲染结果），不缓存原始接口 / DB 数据；
用户页面内跳转前进后退，直接从 Router Cache 拿整页缓存，但无法提取缓存里的原始数据给 Client 组件二次使用，所以不能靠 Router Cache 实现 Client 拿 RSC 原始数据。
一句话面试总结
传 Props/Context：复用 Request Memoization (4)+Data Cache (3)
unstable_cache+use ()：复用 Data Cache (3)
请求同项目 API：复用 Data Cache (3)+Full Route Cache (2)
原生浏览器跨域 fetch：四层缓存全不命中，无法复用 RSC 数据
