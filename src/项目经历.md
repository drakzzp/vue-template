跨屏联动方案解决大屏间跨终端通信

用户在 A 大屏操作（点击地图 Tab、切换下拉、选中图表区块）：
画布组件（Echarts/DataV）绑定click/change原生事件；
前端封装上面标准消息体，填入操作参数payload；
通过当前页面已建立的 WebSocket 长连接，单发消息到服务端。

服务端读取消息里groupId、targetScreenIds；
遍历分组会话池，匹配目标屏幕连接；
原样 JSON 消息推送到所有匹配终端；无关屏幕完全收不到数据。

B 大屏 WebSocket 实例监听onmessage回调，拿到标准 JSON；
按eventType分发到对应业务处理器：
chartClick：用 payload 筛选条件重新请求接口，调用echarts.setOption()重绘图表；
tabSwitch：切换画布 Tab 页面、显隐组件；
alertTrigger：弹出告警弹窗、播放提示动画；
全程无页面刷新，增量更新可视化组件状态。

性能优化/遇到的问题：

部分第三方子组件内部也引入了 echarts，externals 配置不全导致包体积膨胀，大屏首屏加载慢。
Webpack 强制把react、react-dom、echarts、datav全部设为外部依赖，绝不打进组件 umd；平台渲染模板预先全局注入 CDN 版 React、ECharts，挂载到 window，组件直接取用全局变量；
ECharts 实例复用 & 懒渲染（可视化必问优化）

视口外组件延迟初始化（出现在视野里才渲染）
使用 IntersectionObserver 监听组件 DOM 是否进入视口，
没进入视口时只渲染占位 div，不初始化 ECharts / 复杂组件，
进入视口才真正执行初始化，减少首屏压力。

跨屏通信我做了防抖、消息去重、轻量化推送，避免消息风暴，保证弱网环境下多屏同步依然稳定流畅。

组件 UMD 依赖外置，抽离 React/ECharts 用 CDN 全局加载，减少包体积提升加载速度。
ECharts 实例复用、懒渲染、resize 节流，避免大量图表卡顿和内存泄漏。
Schema 动态表单路径级局部渲染，避免整树刷新，提升编辑流畅度。
渲染引擎同构 + 全局缓存，让预览发布秒开。  我对渲染内核、组件、配置做了多层缓存策略，开启 gzip 压缩，实现预览发布秒开。
WebSocket 跨屏消息防抖、去重、轻量化推送，避免消息风暴，保证多屏同步稳定。
图片资源懒加载、压缩、WebP 优化。

就可能会问一下webpack打包，性能优化之类的东西

webpack打包优化
1. externals 抽离公共依赖（最核心、最加分）
2. 代码分割 Code Splitting / 路由懒加载
3. 压缩优化：terser-webpack-plugin 压缩 JS
4. 图片资源优化：asset/resource + 小图片转 base64
5. CSS 抽离与压缩 mini-css-extract-plugin
6. 持久化缓存：filename 加 contenthash
7. DLL 预编译公共依赖（高阶加分）
8. tree-shaking 剔除无用代码

因为我们项目是低代码大屏 + 组件工厂 UMD 打包，所以我还针对组件做了独立构建、按需引用、依赖共享，让每个组件都能最小化打包，
平台运行时动态加载，整体性能提升非常明显。

react的优化
1. 避免不必要重渲染（最核心，适配画布、配置面板大量组件场景）
    函数组件：React.memo + useMemo + useCallback 三件套
    类组件：shouldComponentUpdate / PureComponent
    拆分组件粒度

2. 大数据 & 长列表虚拟滚动（适配配置面板多系列、多数据源列表）
3. 视口懒渲染（你前面实现的 ECharts 延迟初始化）
4. 状态管理优化

5. DOM 与渲染层优化
key 规范：遍历组件、列表不用 index 做 key，用组件唯一 id（componentId），避免 DOM 错位、频繁卸载重建；
批量更新：React18 自动批处理同步 setState；异步场景可用unstable_batchedUpdates合并多次状态修改，减少渲染次数；
频繁 DOM 操作防抖节流：画布 resize、图表拖拽、窗口缩放 30ms 节流，防止高频重绘 ECharts。

前端就是在coze平台上面搭建了个智能回复助手，因为是零代码低代码配置搭建比较灵活，底层llm大模型就是用的豆包大模型
意图识别
可视化流程编排 通过coze拖拽式流程

这个项目底层 AI 流程、拖拽编排画布都是 Coze 平台现成能力，我主要做面向跨境卖家的业务管理后台：一是开发聊天对话界面，对接 Coze 对话 API 实现流式 AI 回复、多轮上下文对话、语音输入；二是做知识库管理页面，对接 Coze 知识库接口完成文档同步索引；三是数据看板、账号权限、国际化整套后台页面，同时做了虚拟列表、长连接稳定、打包优化这类前端工程优化，商家在 Coze 里搭好回复流程，在我们自己的后台接待买家咨询、查看运营数据。

这个 AI 电商回复助手我两头都负责：首先我在 Coze 平台从零搭好了整套智能客服机器人，用它自带的拖拽流程编排，配置电商各类询盘意图、分支逻辑，搭建 RAG 知识库，调豆包大模型的回复风格适配海外外贸场景，并且开通调试好 Coze 开放 API；然后我独立用 React 写了一套给商家用的管理后台，有聊天接待界面、知识库同步页面、数据看板这些，后台通过接口对接我在 Coze 搭好的机器人，商家在我们自己的系统里接待客户、管理知识库、看咨询数据，不用反复切 Coze 平台，实现轻量化落地使用。

难点：
1. 跨境电商多场景意图复杂，分支流程难梳理
海外买家提问句式口语化、中英文混杂、问题嵌套（比如 “多久发货 + 能不能降价 + 破损怎么赔” 三合一提问）；
难点：要拆分几十种核心意图（物流、价格、库存、售后、退换、关税、产品参数等），在 Coze 拖拽里设计多层条件分支、循环对话，还要设置兜底模糊匹配规则，避免机器人识别跑偏答非所问；多轮对话上下文记忆边界不好把控，容易出现聊着聊着遗忘前文需求。

项目难点分两块，一是 Coze 搭机器人的时候，海外买家问题杂、中英文混问，要拆大量意图做多层分支，还要反复调知识库检索和模型指令，防止 AI 乱编答案；二是前端对接流式接口，海外网络差要做好长连接稳控和打字机性能；另外知识库同步有延迟、海量对话列表卡顿、多语言权限适配、全链路排错麻烦都是主要攻坚点。

你给豆包大模型写系统提示词：强制优先引用知识库内容、禁止编造无依据信息、引用资料标注来源；如果不做这条约束，AI 很容易脱离知识库乱回答，这是智能体搭建核心优化点。
