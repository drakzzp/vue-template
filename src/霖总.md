         ```js
            function deepClone(obj){
               function isObject(o){
                  return (typeof o === 'object' || typeof o === 'function') && o !== null
               }
               if(!isObject(obj)){
                  throw new Error('非对象')
               }
               let isArray = Array.isArray(obj)
               let newObj = isArray ? [...obj] : { ...obj }
               Reflect.ownKeys(newObj).forEach(key => {
                  newObj[key] = isObject(obj[key]) ? deepClone(obj[key]) : obj[key]
               })
               return newObj
            }
         ```

   ### 3. proxy -  1.defineProperty只能监听某个属性；2.不会污染原对象

   简易解答：
   Proxy
    - 直接监听对象而非属性
    - 直接监听数组
    - 有多达 13 种拦截方法,不限于 apply、ownKeys、deleteProperty、has 等等是 Object.defineProperty 不具备的
    - Proxy 返回的是一个新对象,我们可以[只操作新的对象]达到目的,而 Object.defineProperty [只能遍历对象属性直接修改]
    
   Object
    - 兼容性好，支持 IE9，而 Proxy 的存在浏览器兼容性问题 (Vue不支持IE8及以下，Vue3不支持IE11)

   详细解答：
   1. 区别一：
      defineproperty[只能监听某个属性]不能全对象监听
      proxy直接监听对象、数组
      
      defineproperty监听需要知道那个对象的那个属性，而proxy只需要知道那个对象就可以了。也就是会省去for  in 循环提高了效率

   2. 区别二：
      proxy不需要借助外部value，也有[单独相配的对象即Reflect]，
      eg：var ob={a:1,b:2}

      在proxy的get里面有target，key，receiver三个值，其中target是对象ob，key是ob.a，receiver是当前代理对象，set里面除了这三个额外多加了一个value，value是传出来的新值。
      所以在get里return的就是target[key]，set里面return的是target[key]=value或者用proxy里的Reflect.set(target，key，value）这样写优雅一点

   + 两对象中get第三个参数receiver
      proxy是 当前代理对象
      Reflect是 target对象中getter调用时的this值

   3. 区别3：[不会污染原对象]（关键区别）

      proxy去代理了ob，他会返回一个新的代理对象不会对原对象ob进行改动，

      defineproperty是去修改原对象，修改原对象的属性，
      而proxy只是对原对象进行代理并给出一个新的代理对象

```js
function throttle(fn, delay){
    let timer = null;
    let startTime = Date.now();
     
    return function(){
        //闭包
        let curTime = Date.now();
        let remaining = delay - (curTime - startTime);  
        let context = this;
        let args = argsment;
        clearTimeout(timer);
        if(remaining <= 0){

            startTime = Date.now();
        }else{
            timer = setTimeout(fn, remaining);
        }
    }
}
```

      ``` -->
# 十一 浏览器缓存机制 开始细看11.30

   1. 概念

   缓存：显著减少网络传输所带来的损耗

   对于一个数据请求包括三步骤：网络请求、后端处理、浏览器响应。

   浏览器缓存能帮助我们优化 一 三 步骤

   ## 浏览器缓存机制：
   [缓存位置]
   [缓存策略]
   [实际场景应用缓存策略]

   ### 1.缓存位置

      分为四种，各有优先级，当依次查找且都没有命中时，才会去请求网络

      1. Service Worker  https,缓存久
      2. Memory Cache    读取快，缓存短
      3. Disk Cache      读取慢，容量、存储时效高
      4. Push Cache      h2,缓存短
      5. 网络请求
   
   #### 具体
      1. Service Worker

         1. 它可以让我们自由控制缓存哪些文件、如何匹配缓存、如何读取缓存，并且[缓存是持续性的]。

         2. 当 Service Worker 没有命中缓存的时候，我们需要去调用 fetch 函数获取数据。
         也就是说，如果我们没有在 Service Worker 命中缓存的话，会根据缓存查找优先级去查找数据。但是不管我们是从 Memory Cache 中还是从网络请求中获取的数据，浏览器都会显示我们是从 Service Worker 中获取的内容。
         (待补充  再看看 上一句的意思)

      2. Memory Cache 

         内存中的缓存

         1. 内存缓存虽然[读取高效]（比磁盘读取快），可是[缓存持续性很短]，会随着进程的释放而释放。关闭Tab页面，内存中的缓存就被释放。
         2. 能不能把数据存放内存中？ 不能。

            > 对于大文件来说，大概率是不存储在内存中的，反之优先
            > 当前系统内存使用率高的话，文件优先存储进硬盘

      3. Disk Cache

         硬盘中的缓存，读取速度慢点，但是都能存。
         比之 Memory Cache [胜在容量和存储时效性上]。

         1. 在所有浏览器缓存中，Disk Cache 覆盖面基本是最大的。
         2. 它会根据 HTTP Header 中的字段判断哪些资源需要缓存，哪些资源可以不请求直接使用，哪些资源已经过期需要重新请求。
         3. 并且即使在跨站点的情况下，相同地址的资源一旦被硬盘缓存下来，就不会再次去请求数据。

      4. Push Cache

         HTTP/2中的内容
         1. 并且[缓存时间也很短暂]，只在会话（Session）中存在，一旦会话结束就被释放。
         2. 几个结论

            > 所有的资源都能被推送，但是 Edge 和 Safari 浏览器兼容性不怎么好
            > 可以推送 no-cache 和 no-store 的资源
            > 一旦连接被关闭，Push Cache 就被释放
            > 多个页面可以使用相同的 HTTP/2 连接，也就是说能使用同样的缓存
            > Push Cache 中的缓存只能被使用一次
            > 浏览器可以拒绝接受已经存在的资源推送
            > 你可以给其他域名推送资源

      5. 网络请求


   ### 2.缓存策略

      强缓存不发送请求到服务器，有时资源更新了浏览器不知道
      协商缓存发送请求到服务器，资源是否更新浏览器知道


   #### 1.强缓存

      强缓存可以通过设置两种 HTTP Header 实现：[Expires] 和 [Cache-Control] 。
      强缓存表示在缓存期间不需要请求，state code 为 200。

   ##### Expires
      
      Expires: Wed, 22 Oct 2018 08:41:00 GMT
      
      1. HTTP/1的产物，表示在上面时间后过期，需要再次请求。 

      2. [受限于本地时间]，修改了本地时间，缓存可能会失效。

   ##### Cache Control

      Cache-control: max-age=30

      1. HTTP/1.1, [优先级高于Expires],表示资源会在30s后过期，需要再次请求。

      2. Cache-Control 可以[在请求头或者响应头中设置],可配合多个指令一起使用

         1. no-cache：需要进行协商缓存，发送请求到服务器确认是否使用缓存。
         2. no-store：禁止使用缓存，每一次都要重新请求数据。
         3. public：可以被所有的用户缓存，包括终端用户和 CDN 等中间代理服务器。
         4. private：只能被[终端用户的浏览器缓存]，不允许 CDN 等中继缓存服务器对其缓存。


      img:https://img-blog.csdnimg.cn/img_convert/a866728a7fcdee6444be4a564f57766b.png


   #### 2.协商缓存

      如果缓存过期了，就需要发起请求验证资源是否有更新。协商缓存可以通过设置两种 HTTP Header 实现：[Last-Modified] 和 [ETag] 。

      当浏览器发起请求验证资源时，如果资源没有做改变，那么服务端就会返回 304 状态码，并且更新浏览器缓存有效期。

   ##### Last-Modified 和 If-Modified-Since

      1. Lash-Modified 表示本地文件最后修改日期。
      
      2. If-Modified-Since 会将 Last-Modified 的值发送给服务器，询问服务器在该日期后资源是否有更新？

         有更新：将新的资源发送回来
         无更新：返回304状态码

      弊端：
         1. 如果[本地打开缓存文件，即使没有对文件进行修改]，但还是会造成Last-Modified被修改，服务端不能命中缓存[导致发送相同的资源]
         2. 因为Last-Modified只能以秒即使，如果[在不可感知的时间内修改完成文件]，那么[服务端会认为资源还是命中了，不会返回正确的资源]

      解决: ETag
      
   ##### ETag 和 If-None-Match

      1. Etag类似于文件指纹，If-None-Match 会将当前 ETag 发送给服务器，询问该资源 ETag 是否变动，有变动的话就将新的资源发送回来。

      2. ETag [优先级比Last-Modified高]。

   Question: 如果什么缓存策略都没设置，那么浏览器会怎么处理？

   对于这种情况，浏览器会采用一个启发式的算法，通常会 [取响应头中的 Date 减去 Last-Modified 值 的10%]作为缓存时间。 
   （比如响应头时间是10.11，Last-Modified是10.01，那么相减得到10天，1天作为缓存时间）


   ### 3. 实际场景应用缓存策略

      1. 频繁变动的资源

      Cache-Control: no-cache 使浏览器每次都请求服务器，然后配合 ETag 或者 Last-Modified 来验证资源是否有效。
         
      这样的做法虽然不能节省请求数量，但是能显著减少响应数据大小。

      2. 代码文件（除html外的代码文件，因为HTML文件一般不缓存或缓存时间很短）

      一般来说，现在都会使用工具来打包代码，那么我们就可以[对文件名进行哈希处理]，只有当代码修改后才会生成新的文件名。  eg. index.3ujfhog.js

      基于此，我们就可以给代码文件设置缓存有效期一年 Cache-Control: max-age=31536000，这样只有当 HTML 文件中引入的文件名发生了改变才会去下载最新的代码文件，否则就一直使用缓存。

   ### 4.状态码

      301 永久重定向，可缓存
      302 临时重定向
    ```

   ### 描述
   1. 递归[自顶向下]：爬5级，如果爬1级，剩下4级要爬；所以拆分问题，爬4级有几种方式， 则爬5级= 4级+3级
                 如果爬2级，剩下3级要爬             爬3级有几种方式
      复杂度：最高
   2. DP[自底向上]：爬到i层，必须先爬到i-2 或 i-1层
      先创建一个长度n+1的数组，dp[1]=1;dp[2]=3;
      做个for循环，从i=3开始，dp[i]=dp[i-1]+dp[i-2],再return dp[n]
      复杂度：记录了所有的状态
      ```js
      let climbStairs = (n) => {
         let dp = new Array(n+1).fill(null);
         dp[0] = 0;
         dp[1] = 1;
         dp[2] = 2;
         for(let i=3;i<dp.length;i++){
            dp[i] = dp[i-1]+dp[i-2]
         }
         return dp[n]
      }
      ```
   3. 斐波那契
   压缩空间，优化: [dp[i] 只与过去的两项：dp[i-1] 和 dp[i-2] 有关，没有必要存下所有计算过的 dp 项。用两个变量去存这两个过去的状态就好。]
   ```js
      <!-- 爬楼梯 -->
      var fib = function(n){
         if(n==1) return 1
         if(n==2) return 2
         
         let first = 1
         let second = 2
         for(let i=3;i<=n;i++){
            let third = first + second;
            first = second;
            second = third;
         }
         return second
      }
   ```

1. 前端埋点方案

      代码埋点
         嵌入代码的形式，eg.监控用户的点击事件
         优点：在任意时刻，精确发送或保存所需的数据信息
         缺点：工作量较大，每一个组件的埋点都需要添加相应的代码
      可视化埋点
         用可视化手段，将业务代码和埋点代码分离
         缺点：埋点的控件有限，不能自定义
      无埋点
         全部埋点，前端的任意事件都绑定一个标识，所有事件都记录下来。
         优点：采集全量数据，不会遗漏
         缺点：加重服务器压力，不能灵活定制

1. doctype作用：
   严格模式(标准模式)    --    混杂模式(怪异/兼容模式)

    1. TCP 三次握手

        张三招手（SYN） --- 李四微笑（ACK）后招手（SYN） --- 张三微笑（ACK） ---
                   张三syn_sent                   张三estalished
                     李四syn_rcvd                                  李四estalished     

        详细版： Initial Sequence Number：初始化序列号

        0.    刚开始：客户端处于closed，服务端处于listen
        1. 第一次握手：客户端给服务端发送一个SYN报文，并指明客户端的初始化序列号ISN(c)。此时客户端处于 SYN_SEND

            SYN=1,初始序号seq=x

        2. 第二次握手：服务端收到客的SYN后，会以自己的SYN作为应答，并指明自己的ISN(s),同时会把客户端的 ISN + 1 作为ACK的值，表示自己已经收到了客的SYN。此时服务器处于 SYN_RCVD

            在确认报文段中SYN=1，ACK=1，确认号ack=x+1，初始序号seq=y

        3. 第三次握手：客户端收到SYN后，会发送一个ACK，把服务端的ISN + 1作为ACK的值，表示已经收到服的SYN。客户端此时处于ESTABLISHED。此时，双方已建立起了连接。

            确认报文段ACK=1，确认号ack=y+1，序号seq=x+1
        
        [Question]
        Q1：为什么不能是两次？
        客户端第一次发起请求 - 遇到网络问题，客户端就发第二次 - 第二次收到确认建立了连接，数据传输完毕，释放连接 - 第一次因网络结点滞留，延误后到达服务端 - 服务端以为客户端新的请求，就会发出确认报文 - 客户端忽略了，服务端一直等待响应，浪费资源
      （另一个解释 可看）
        Q2:第三次握手的时候建立起连接了，是可以携带数据的。第一次、第二次握手不可以携带数据,容易受攻击。

    2. 四次挥手

        张三挥手（fin） -- 李四伤感微笑（ack） -- 李四挥手（fin） -- 张三伤感微笑（ack）

9. 组件中 data 为什么是一个函数？[好问题]

   组件中的 data 必须是一个函数，然后 return 一个对象，而 new Vue 实例里，data 可以直接是一个对象

   - 因为组件是用来复用的，且 JS 里对象是引用关系，
   - 如果组件中 data 是一个对象，那么这样[作用域没有隔离]，子组件中的 data 属性值会相互影响
   - 如果组件中 data 选项是一个函数，那么每个实例可以维护一份被返回对象的独立的拷贝，组件实例之间的 data 属性值不会互相影响
   
   - 而 new Vue 的实例，是不会被复用的，因此不存在引用对象的问题。

12. Vuex

    每个Vuex应用的核心就是store（仓库），即 容器，包含着大部分的状态（state）
    
    - 它的状态存储是响应式的。 当组件从store中读取状态时，若store状态发生变化，那相应的组件也会更新
    - 改变store中的状态：显式地提交（commit） mutation.  方便跟踪每个状态的变化

    Vuex 解决了[多个视图依赖于同一状态]和[来自不同视图的行为需要变更同一状态]的问题，开发者聚焦于数据的更新而不是数据在组件之间的传递上

    模块：
    1. State: 定义了应用状态的数据结构，可以在这设置初始状态
    2. Getter: 允许组件从 Store 中获取数据， mapGetters 辅助函数仅仅是将Store中的 getter 映射到局部计算属性
    3. Mutation： 唯一更改 store 中状态的方法，必须是[同步函数]
    4. Action: 用于提交 mutation，而不是直接变更状态，可以包含任意异步操作
    5. Module: 允许将单一的store拆分为多个store且同时保存在单一的状态树中

    state：用于数据的存储，是store中的唯一数据源
    getters：如vue中的计算属性一样，基于state数据的二次包装，常用于数据的筛选和多个数据的相关性计算
    mutations：类似函数，改变state数据的唯一途径，且不能用于处理异步事件
    actions：类似于mutation，用于提交mutation来改变状态，而不直接变更状态，可以包含任意异步操作
    modules：类似于命名空间，用于项目中将各个模块的状态分开定义和操作，便于维护

    ```js
        }

    }
    ```

## 二. 平时有关注过前端的内存处理？

1. 内存的生命周期

    内存分配：当我们在声明变量、函数、对象时，js会自动分配内存
    内存使用：即读写内存，调用、使用 变量 函数的时候
    内存回收：使⽤完毕，由垃圾回收机制⾃动回收不再使⽤的内存

2. js中的垃圾回收机制

    1. 引用计数垃圾回收

    "引用":   a对象对b对象有访问权限，那么 a引用b  （例如，⼀个Javascript对象具有对它原型的引⽤（隐式引⽤）和对它属性的引⽤（显式引⽤）。）

    缺陷：循环引用,a 引用 b,b 引用 a,尽管他们已不再使用，但就不会回收，导致内存泄漏。

    2. 标记清除算法

    “无法达到的对象"

    从根部（JS中即全局对象）出发,所有都标记，达到的先清除，剩下还有标记的，稍后进行回收。

    1. 垃圾收集器在运⾏的时候会给存储在内存中的所有变量都加上标记。
    2. 从根部出发将能触及到的对象的标记清除。
    3. 那些还存在标记的变量被视为准备删除的变量。
    4. 最后垃圾收集器会执⾏最后⼀步内存清除的⼯作，销毁那些带标记的值并回收它们所占⽤的内存空间。

3. 内存泄露 

    1. 全局变量

    window.u = 'sss'
    window.u = null (用完了要赋值null)

    2. 未被清理的定时器和回调函数

    3. 闭包
    
    >简单来说，闭包：一个内部函数，有权访问包含其的外部函数中的变量。

    >mdn定义：一个函数和对其周围状态（lexical environment，词法环境）的引用捆绑在一起（或者说函数被引用包围），这样的组合就是闭包（closure）。也就是说，闭包让你可以在一个内层函数中访问到其外层函数的作用域。 
    在 JavaScript 中，每当创建一个函数，闭包就会在函数创建的同时被创建出来。 
    （js本身就有一个主函数的概念）
    
    4. dom引用
    ```JS
    const element = {

    document.body.removeChild(document.getElementById('image'))
    // 即使移除了元素，对image的引用还是存在
    ```

   11. 怎样实现一个promise.all

   ```js
      function PromiseAll(pArr) {
         return new Promise((resolve, reject)=> {
            let arr = []
            let i = 0

            function handleData(index, data) {
               arr[index] = data
               i++;
               if( i === pArr.length ) {
                  resolve(arr)
               }
            }

            for( let i=0, i < pArr.length; i++){
               pArr[i].then((data)=> {
                  handleData(i, data)
               }, reject)
            }
         })
      }
   ```
